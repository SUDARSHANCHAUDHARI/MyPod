Origin

Investigated github.com/stablyai/orca — an 11.8k-star MIT-licensed Electron app ("AI Orchestrator") that runs multiple coding agents (Claude Code, Codex, Cursor, etc.) in parallel, each in its own isolated git worktree.

We are not cloning Orca's code or building a competing product — copying one specific engineering pattern and adapting it into existing SudarshanTechLabs autonomous-task tooling.

Current State (before MyPod)

  • ~/.claude/autonomous.sh + RALPH_PROMPT.md run CLAUDE_TASKS.md tasks sequentially, one at a time, in a single working directory.
  • dispatching-parallel-agents skill exists for firing multiple agents at once, but isn't wired into the autonomous task queue.
  • Claude Code's Agent tool already supports isolation: "worktree" — it can create an isolated git worktree per agent call automatically.

The Gap

No safe, race-free worktree lifecycle management for running several autonomous tasks at once against the same repo. Orca's src/main/git/worktree.ts solves exactly this:

1. Creation — git worktree add --no-track -b <branch> (never inherits the base branch's upstream, so git status doesn't misreport "behind" on an unpublished branch), plus setting push.autoSetupRemote=true once so the agent's first git push just works.
2. Removal — delete the branch with -d (safe, refuses unmerged commits) by default. Only use -D (force) to roll back a worktree whose creation just failed — never to clean up a worktree that might contain real agent work. Falls back to a squash-merge detection check (rev-list/merge-base) before allowing a delete that -d would otherwise refuse.
3. Concurrency — dedupe concurrent git worktree list scans per repo path (in-flight promise map) — matters once several parallel agents poll status on the same repo at once.

Recommendation

Rewrite autonomous.sh to fan out N pending CLAUDE_TASKS.md tasks across N parallel Agent tool calls (isolation: "worktree") instead of one sequential claude --print loop, applying Orca's three safety rules above to whatever wrapper manages worktree create/remove.

Naming

Project named MyPod — a "pod" of parallel agents, kept distinct from Orca's own branding. Repo scaffolded at AIProjects/MyPod/.

Implementation Notes

built and smoke-tested

No Bash ≥ 4 requirement, after all. The plan assumed wait -n would need a Homebrew Bash install. Checked instead: this machine has only stock Bash 3.2, no Homebrew bash. mypod.sh dispatches jobs and waits on the explicit pid list instead — fully portable to 3.2, no version bump needed.
No timeout/gtimeout either — genuinely absent on this machine, not just a Linux-only assumption. lib/run-task.sh implements its own timeout: run the command in its own process group (set -m) and kill the negative PID on timeout, which reaps any children the agent spawns too, not just the top-level process.
Bug found during smoke testing: the no_commits check used git log ... | grep -q . in a script with set -o pipefail. grep -q closes its input on first match; if git log still had more to write, it got SIGPIPE, and pipefail reported the whole pipeline as failed — misclassifying a task with real commits as no_commits and discarding its worktree. Fixed with git rev-list --count (no pipe, nothing to race). The test suite had the same anti-pattern in four assertions — fixed the same way there.
Not done: not wired into ~/.claude/autonomous.sh — runs standalone via bash mypod.sh. That integration is a deliberate follow-up once it's proven itself in real use.
